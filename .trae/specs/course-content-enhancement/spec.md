# 课程内容完善 - Product Requirement Document

## Overview
- **Summary**: 基于《市场数据分析》课程详细教学设计，完善现有在线学习平台的课程内容体系。将6个阶段54课时的完整教学设计落地到 course-content.json、questions.json 和样本数据中，实现"裸练→辅助→对比反思"三阶AI使用原则，以及分级验收标准（A/B/C三级）。
- **Purpose**: 解决当前课程内容过于简化、缺乏教学设计梯度、AI使用原则不明确、验收标准不清晰等问题，使平台内容与高职院校教学大纲完全对齐。
- **Target Users**: 高职院校电子商务/市场营销专业教师与学生

## Goals
- 完善6个阶段共54课时的完整课程内容，与教学设计文档100%对齐
- 落地"裸练→辅助→对比反思"三阶AI使用原则，每课时明确标注AI状态
- 建立A/B/C三级分级验收标准体系
- 补充变式迁移练习环节（阶段2、3、4）
- 完善阶段4的4个业务场景深度分析（流量转化、用户价值、竞品市场、活动复盘）
- 补充阶段5的数据看板与报告撰写内容
- 补充阶段6的汇报与考核内容
- 完善题库，每个阶段至少15道题，覆盖所有知识点
- 补充样本数据（8月对比数据、厨房收纳子品类数据、双11历史数据等）

## Non-Goals (Out of Scope)
- 不修改前端UI和交互逻辑
- 不修改后端（无后端）架构
- 不新增功能模块（如实时协作、视频播放等）
- 不做移动端适配优化
- 不涉及评分系统的自动化（仍由教师人工评定）

## Background & Context
当前平台已有基础框架和6个阶段的简化内容，但与完整的教学设计相比存在以下差距：
1. 课时数量不足：阶段2只有10课时（应为12课时），阶段4只有约8课时（应为16课时）
2. AI使用原则不明确：没有明确的"AI禁用/AI可用"环节划分
3. 缺乏变式迁移环节：教学设计要求阶段2、3、4各有变式迁移专项
4. 验收标准不清晰：只有简单的rubric criteria，缺乏A/B/C三级标准
5. 阶段4场景不完整：缺少用户价值、竞品市场、活动复盘等场景
6. 阶段5内容单薄：看板制作和报告撰写内容不足
7. 阶段6几乎空白：汇报考核环节缺失
8. 题库量不足：每个阶段只有约10道题

## Functional Requirements
- **FR-1**: 完善阶段1（入职培训与项目启动）6课时内容，补充双人协作分工、工具环境准备等内容
- **FR-2**: 完善阶段2（数据采集与标准化清洗）12课时内容，新增含陷阱数据集练习、变式迁移环节
- **FR-3**: 完善阶段3（核心指标核算与异常初判）12课时内容，新增手写口径练习、含陷阱任务、变式迁移
- **FR-4**: 完善阶段4（核心业务场景专项深度分析）16课时内容，覆盖流量转化、用户价值、竞品市场、活动复盘4个场景，每个场景4课时含变式迁移
- **FR-5**: 完善阶段5（数据看板搭建与分析报告撰写）12课时内容，补充看板设计、报告写作、交叉评审等
- **FR-6**: 完善阶段6（项目汇报与转正考核）6课时内容，补充路演评分、终结考核等
- **FR-7**: 每课时明确标注AI状态（AI禁用/AI可用），在UI中清晰展示
- **FR-8**: 每个阶段建立A/B/C三级分级验收标准
- **FR-9**: 完善题库，每个阶段至少15道题，覆盖单选、多选、填空、实操题
- **FR-10**: 补充必要的样本数据文件（8月对比数据、厨房收纳子品类、双11数据等）

## Non-Functional Requirements
- **NFR-1**: 内容完整性：6个阶段54课时与教学设计文档100%对应
- **NFR-2**: 数据一致性：所有样本数据的业务口径统一，前后阶段数据可复用
- **NFR-3**: JSON格式正确：所有数据文件语法正确，可正常解析加载
- **NFR-4**: 兼容性：不破坏现有页面的正常加载和使用
- **NFR-5**: 可维护性：课程内容结构清晰，便于后续教师自行修改补充

## Constraints
- **技术约束**: 纯静态JSON数据，不能有动态生成逻辑；使用现有的course-content.json数据结构
- **业务约束**: 必须基于"杭州悦享家居用品有限公司"的业务背景，数据口径需前后一致
- **数据约束**: 所有样本数据必须是脱敏的模拟数据，不能使用真实企业数据
- **课时约束**: 总课时必须严格等于54课时，各阶段课时数与教学设计一致

## Dependencies
- 现有 course-content.json 数据结构
- 现有 questions.json 数据结构
- 现有 sample-data/ 目录下的样本数据
- learn.html、practice.html、test.html 等页面的渲染逻辑

## Assumptions
- 现有页面能够正确渲染新增的课时类型和内容字段
- AI状态标记（aiEnabled字段）已有前端支持，仅需确保数据正确设置
- 分级验收标准可通过现有的submission和rubric字段扩展实现
- 变式迁移练习可使用现有的variant类型课时
- 学生以个人为单位使用平台，双人协作仅在内容中体现要求，不做技术层面的协作功能

## Acceptance Criteria

### AC-1: 课时数量与结构对齐
- **Given**: 课程内容已完善
- **When**: 加载 course-content.json
- **Then**: 6个阶段总课时为54课时，各阶段课时数为：阶段1=6、阶段2=12、阶段3=12、阶段4=16、阶段5=12、阶段6=6
- **Verification**: `programmatic`
- **Notes**: 可通过JSON解析统计各阶段lessons数量验证

### AC-2: AI使用原则明确标注
- **Given**: 课程内容已完善
- **When**: 查看任意课时
- **Then**: 每个课时都有明确的aiEnabled字段（true/false），且与教学设计中的AI状态一致
- **Verification**: `programmatic`
- **Notes**: AI禁用环节（裸练）aiEnabled=false，AI可用环节aiEnabled=true

### AC-3: 分级验收标准完整
- **Given**: 课程内容已完善
- **When**: 查看有提交要求的课时
- **Then**: rubric中包含A/B/C三级验收标准的具体描述
- **Verification**: `human-judgment`
- **Notes**: C级=达标，B级=良好，A级=优秀，每级有明确可衡量的标准

### AC-4: 变式迁移环节到位
- **Given**: 课程内容已完善
- **When**: 查看阶段2、3、4
- **Then**: 阶段2有2课时变式迁移，阶段3有2课时变式迁移，阶段4有4课时变式迁移（每个场景1课时）
- **Verification**: `programmatic`
- **Notes**: 变式迁移课时应有明确的"跨品类/跨时段"验证设计

### AC-5: 阶段4四个场景完整
- **Given**: 课程内容已完善
- **When**: 查看阶段4
- **Then**: 包含4个场景：流量与转化归因分析、用户价值与复购分析、竞品与市场趋势分析、营销活动效果复盘，每个场景4课时
- **Verification**: `programmatic`
- **Notes**: 每个场景都包含理论、跟练、独立实操、变式迁移4个环节

### AC-6: 题库充足且覆盖全面
- **Given**: 题库已完善
- **When**: 查看每个阶段的测试题
- **Then**: 每个阶段至少15道题，题型包含单选、多选、填空，覆盖该阶段所有核心知识点
- **Verification**: `programmatic`
- **Notes**: 难度分布合理，基础题60%、中等题30%、提高题10%

### AC-7: 样本数据完备
- **Given**: 样本数据已补充
- **When**: 查看sample-data目录
- **Then**: 包含订单数据（9月全量、8月对比、变式迁移用）、流量数据、用户数据、竞品数据、活动数据等
- **Verification**: `programmatic`
- **Notes**: 所有数据文件格式正确（JSON），业务口径统一

### AC-8: 不破坏现有功能
- **Given**: 课程内容已完善
- **When**: 在浏览器中访问学习页面、实操页面、测试页面
- **Then**: 所有页面能正常加载，无JS错误，功能正常使用
- **Verification**: `programmatic`
- **Notes**: 使用浏览器自动化测试验证

## Open Questions
- [ ] 分级验收标准（A/B/C）的具体呈现形式：是仅在内容中文字描述，还是需要在UI中做专门的展示组件？
- [ ] 变式迁移练习的提交和评分：与普通练习一样，还是有专门的对比反思提交？
- [ ] 阶段6的终结考核：是使用现有的测试功能，还是需要单独的"限时实操"模式？
- [ ] 双人协作：是否需要在平台中体现小组功能，还是仅在内容中说明由学生线下组队？
